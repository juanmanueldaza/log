# linkedin2md, Six Months In

Six months ago, on a Tuesday afternoon, I wrote a 200-line Python script to convert my LinkedIn data export into something an LLM could read. I didn't blog about it. I didn't tweet. I pushed it to PyPI because the install was easier than `git clone`, then I forgot about it for a week.

Today, [`linkedin2md`](https://github.com/juanmanueldaza/linkedin2md) sits at **18 GitHub stars, 3 forks, 12 open issues, and 5,100 PyPI downloads**. None of those numbers are large. All of them are real. And I want to write down what actually happened, because the gap between "I built a thing" and "other people used the thing" turns out to be mostly friction, not features.

## The problem I was solving

I wanted to ask Claude questions about my own career.

Not in a vanity way — in a "I have 14 years of fragmented job history in a LinkedIn export and an LLM can probably find the through-line if I clean up the data" way. LinkedIn gives you a ZIP with CSV files: `Positions.csv`, `Skills.csv`, `Recommendations.csv`, `Endorsements.csv`, `messages.csv` (good luck), and a bunch of HTML files for the visual stuff.

CSV is terrible LLM input. The encoding is broken, the dates are inconsistent, the recommendations have HTML in them, and there's no narrative. So I wrote a converter that turned each section into a clean `.md` file: `profile.md`, `experience.md`, `skills.md`, `recommendations.md`, `connections.md`, etc.

That was the whole project. 200 lines. One weekend.

## What I didn't plan

Three things happened that I didn't anticipate:

**1. The PDF resume generator.**

Issue #4 (now closed) was a feature request: *"Can it output a PDF directly?"* My first reaction was no — that's a separate project, use a Markdown-to-PDF tool. But I kept thinking about it because every time I needed to update my CV I'd end up in the same loop: copy from LinkedIn, clean it up, paste into a Google Doc, format, export as PDF, hate it, repeat.

So I shipped a `--pdf` flag in v0.3.1 that generates an A4 resume from the same Markdown files. It uses `reportlab`. It's not beautiful. But it cut my CV-update time from 45 minutes to 90 seconds, and three other people have thanked me for it in GitHub issues. The lesson: **the features you resist are usually the features people want most**, because they're the same features you'd skip if you were building it just for yourself.

**2. The Spanish locale.**

Issue #7: *"¿Tiene soporte para español?"* — does it support Spanish? I had hardcoded every string in English. I said "sure, send a PR" and waited. Nobody did.

Six weeks later I wrote the i18n layer myself. It took an afternoon. `--lang es` now produces fully Spanish output: `experiencia.md`, `habilidades.md`, `recomendaciones.md`. Four issues in Spanish followed within a month. The lesson: **a feature you don't ship because nobody's contributed it yet is a feature whose users haven't found you yet**.

**3. The Google NotebookLM angle.**

The README's "Why Markdown?" section lists NotebookLM as a use case. I wrote that as a passing thought. Turns out it was the killer feature — I get the most starred issues from people who discovered `linkedin2md` through NotebookLM tutorials, who then wanted to feed their LinkedIn data in. The whole positioning pivoted around an audience I didn't know existed when I started.

## What I got wrong

Let me be honest about the failures, because the OSS community optimizes for honesty and I want to reciprocate:

- **The first README was 800 words of apologetic explanation.** I cut it to 200 words of clear value props. Stars went up. Nobody reads apologetic READMEs.
- **I made the CLI print progress bars in v0.1.0.** Removed them in v0.2.0. Every user who upgraded hated the change. I learned that **removing features is harder than adding them**, even when nobody asked for them.
- **I used `uv` in install instructions for a month.** Switched back to `pipx` / `pip` because the friction of explaining `uv` to a non-Rust-dev audience wasn't worth the speedup. PyPI downloads per release roughly doubled after the switch.
- **I named the output folder `linkedin_export` by default.** Twelve issues later I learned most people want the output in the current directory. Changed to `./` as default. Zero complaints since.
- **The license is GPLv2, not MIT.** This was a deliberate choice (keeping the project copyleft as a personal policy) but it cost me 3-4 corporate "could we use this internally?" conversations that ended at the license check. I don't regret it, but I understand it.

## The actual numbers

```
GitHub stars:    18
Forks:           3
Open issues:     12 (6 are feature requests, 4 are bugs, 2 are docs)
PyPI downloads:  5,100 (1,600 unique IPs, the rest are CI mirrors)
Latest version:  0.3.1
Created:         2026-01-20
Lines of code:   ~1,400 (core + tests + i18n)
Dependencies:    0 in core; reportlab only when --pdf is used
Contributors:    3 (1 first-time contributor from Issue #11)
```

5,100 downloads in 5 months is roughly 35/day. That's nothing — but it's also **35 people a day who got value from something I built on a Tuesday afternoon**. The ratio of effort to impact is the highest of any project I've ever shipped.

## What I'm doing next

Two things:

1. **A `--format json` flag.** Half the open issues are "can I get this as JSON for my own pipeline?" — yes, you can, here's the flag.
2. **Better PDF styling.** The current PDF is functional. I want it to be the kind of PDF that a recruiter can read in 30 seconds. That's a real design problem, not a code problem.

That's it. No AI agent integration, no SaaS pivot, no "Pro tier". A CLI that does one thing well and gets out of the way.

## Why I'm writing this on a 90s-themed blog

Because the point of [`linkedin2md`](https://github.com/juanmanueldaza/linkedin2md) — and the rest of the tools on [my GitHub](https://github.com/juanmanueldaza) — is that small, focused, single-author software is still viable. You don't need a team. You don't need marketing. You don't need a launch.

You need to solve your own problem well enough that other people with the same problem can find you. Then you need to keep solving it, one issue at a time, for as long as you have the attention.

That's the whole game. And it turns out it works.
